# autobug-analyze

Скилл помогает разобрать автобаг: понять причину, решить что с ним делать, и выполнить выбранное действие — не больше, не меньше.

> **TODO / placeholders.** В этом скилле имена MCP и CLI-команд для автобаг-системы и трекера ещё не зафиксированы. Везде, где встречается `<autobug-mcp>` или `<tracker-mcp>`, подставь доступный инструмент из текущей сессии (проверь `mcp__*` в списке инструментов и `gh`/CLI). Если подходящего инструмента нет — явно скажи пользователю и попроси его выполнить шаг вручную, отдав готовый текст/команду.

## Когда включаться

Запускай скилл, если выполнено хоть что-то из:

- пользователь дал URL автобага или его идентификатор,
- пользователь скопировал exception/stack trace и сослался на мониторинг, алерт, прод-инцидент, автобаг,
- пользователь задал триаж-вопрос: «шумит или реально сломано», «что с этим делать», «заведи карточку», «замьють это».

Если на вход пришёл только стек-трейс без ссылки на систему — всё равно можно помочь с диагностикой, но для веток 1 и 3 нужно попросить ID автобага: без него не получится ни понизить severity, ни прилинковать карточку.

## Что пользователь ждёт на выходе

Короткое резюме инцидента → гипотеза о причине → **одна** рекомендуемая ветка с обоснованием → конкретные команды/текст для её выполнения → запрос подтверждения. Не выполняй действия до подтверждения человеком — особенно правки кода и записи в трекер/автобаг-систему.

## Workflow

### 1. Вытащить контекст автобага

Через `<autobug-mcp>` получи по ID/URL:
- текст исключения и сообщение,
- полный stack trace,
- окружение (прод/стейдж, сервис, версия),
- частоту и окно (сколько раз за сутки/час, первое и последнее появление),
- сэмпл входных данных / headers / userId, если доступно.

Если инструмента нет — попроси пользователя вставить этот блок руками. Без контекста дальше не двигаемся: гадать по одному сообщению бессмысленно.

### 2. Найти код

По стеку определи модуль/класс/метод и прочитай код в репозитории (Grep/Read). Важно прочесть:
- саму точку падения,
- ближайший вызывающий код (где рождается аргумент, который упал),
- связанную конфигурацию/DI-регистрацию, если падение похоже на unresolved-зависимость или неправильную настройку.

Если стек указывает на код, которого в текущем репозитории нет (чужой сервис, NuGet-пакет) — скажи об этом и попроси уточнить репозиторий.

### 3. Поставить гипотезу

В 2-4 предложениях: **что** упало, **почему**, **при каких условиях**, **транзиентное ли**. Цель — отделить «наш баг» от «внешнего шума», и узкое место от системной проблемы. Если гипотез несколько — выбери наиболее вероятную и укажи, какой сигнал из контекста подтвердил бы альтернативу.

### 4. Выбрать ветку

Предложи одну ветку. Критерии:

| Ветка | Когда выбирать |
|---|---|
| **1. Мьют / фильтр** | Ошибка транзиентная или внешняя (сетевые таймауты, 5xx от стороннего API, cancelled requests на деплое), уже корректно обработана в коде, но залогирована на Error. Бизнес-воздействия нет. Ретраи/graceful degradation уже работают. |
| **2. Починить по месту** | Причина локальна и очевидна: null-гард, off-by-one, забытый `await`, неверный `ConfigureAwait`, неправильный формат парсинга, неправильный `CancellationToken`, опечатка в константе. Правка ≤ ~30 строк, без миграций и архитектурных изменений. Можно накрыть юнит-тестом. |
| **3. Карточка в трекере + ignore** | Требуется проектирование, миграция данных, обсуждение с другой командой, изменение контракта, или правка затрагивает несколько модулей. Также — когда причина ясна, но исправление нетривиальное и не влезает в текущую сессию. |

Если находишься между двумя ветками — склоняйся к более лёгкой (1 → 2 → 3), но честно проговори сомнение.

### 5. Выполнить ветку — только после подтверждения

Не выполняй действий, пока человек не сказал «да». Показывать надо:
- выбранная ветка и почему,
- конкретные команды/правки, которые будут сделаны,
- что останется за скобками.

## Плейбуки по веткам

### Ветка 1 — мьют / фильтр

Цель: убрать шум из автобаг-системы, не трогая код.

Через `<autobug-mcp>` либо понизь severity (Error → Warning/Info), либо добавь фильтр по сигнатуре (тип исключения + сообщение + endpoint). В комментарии к действию зафиксируй:
- **Почему мьютим** (например: «HttpRequestException к partner-api — транзиент, у нас уже 3 ретрая с exp-backoff»).
- **До какого условия** (частота > X/час, появление нового endpoint — вернуть Error).
- **Когда пересмотреть** (обычно 30 дней).

Не мьють без письменного следа — через месяц никто не вспомнит, почему это шум.

### Ветка 2 — починить по месту

Ограничения скоупа, которые держим жёстко:
- правим только место падения и минимально необходимый контекст;
- не рефакторим соседний код, даже если чешется — это отдельная задача;
- обязательно добавляем тест, воспроизводящий баг **до** правки (AAA-стиль, «should ...», без inline-меток — см. конвенции репо).

Чек-лист перед правкой:
1. Воспроизвёл ли я условие из стека? (В тесте или в голове — но явно.)
2. Понимаю ли я root cause, а не просто «если заткнуть тут — не упадёт»?
3. Могу ли я сформулировать инвариант, который защищает правка? («После этой строки `customer` не может быть null, потому что ...».)

Если на любом пункте «нет» — скорее всего ты в ветке 3, а не 2.

### Ветка 3 — карточка в трекере + ignore

Шаблон текста карточки (всё по-русски, идентификаторы по-английски):

```
# <короткий заголовок: симптом + область>

## Problem
Что происходит с точки зрения пользователя/системы. Ссылка на автобаг.

## Why it matters
Кого задевает (фича, роль пользователя, деньги, SLA).
Как часто воспроизводится, на каком окружении.

## Root cause
Гипотеза причины со ссылкой на файл:строку. Если причина неизвестна — честно
пиши «требуется воспроизведение» и укажи, что нужно собрать.

## Proposed fix
Набросок подхода. Не обязан быть финальным, но должен показать масштаб.

## Risks / out of scope
Что может сломаться при правке. Что сознательно не делаем сейчас.
```

Далее:
1. Через `<tracker-mcp>` заведи карточку, получи её ID/URL.
2. Через `<autobug-mcp>` прилинкуй карточку к автобагу и поставь ignore на N дней (уточни у пользователя; если не сказано — предложи 14 и жди подтверждения).
3. В чате отдай: ссылку на карточку, ссылку на автобаг, срок ignore, дату, когда автобаг снова «оживёт».

Без карточки ignore не ставим: ignore без тикета = тихое забытое.

## Формат ответа в чате

После шага 3-4 (до выполнения) отвечай примерно так:

```
**Summary.** <1 абзац: что упало, где, как часто>.

**Hypothesis.** <2-4 предложения о причине>.

**Рекомендация: ветка N — <название>.** <1-3 предложения обоснования>.

**План действий.**
1. <конкретный шаг с командой/файлом>
2. <следующий>
...

Подтверди, что идём по этому плану?
```

## Чего не делать

- Не править код вне ветки 2 «по дороге».
- Не закрывать автобаг молча («fixed» без ссылки на правку/карточку).
- Не ставить ignore без карточки, даже на пару дней.
- Не гадать, если стек указывает на код вне репозитория — сначала уточни, какой репо открыт/нужен.
- Не выполнять правки/MCP-действия до подтверждения человеком.
