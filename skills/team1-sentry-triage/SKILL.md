# Sentry Triage

## Overview

Workflow для дежурного инженера: от ссылки на автобаг в Sentry до решения (silence / quick fix / тикет+ignore). Каждый триаж завершается ровно одной из трёх рекомендаций, аргументированной данными из Sentry и кодом из репозитория.

## Signature

```
/team1-sentry-triage <sentry-url>
```

`<sentry-url>` — обязательный. Если не передан — **спросить у пользователя** одним вопросом и не продолжать до ответа.

## Preflight — блокирующий чеклист

Выполнить до любой другой работы. Если что-то не ОК — остановиться и попросить пользователя починить.

1. **Sentry MCP подключён.** Проверка: есть ли `mcp__sentry__*` в доступных тулзах. Если deferred — загрузить через `ToolSearch` с `select:mcp__sentry__get_sentry_resource,mcp__sentry__list_issue_events,mcp__sentry__get_issue_tag_values,mcp__sentry__update_issue`. Если Sentry MCP вообще отсутствует — подсказать `/mcp-setup:sentry` и остановиться.
2. **`glab` установлен и авторизован.** Проверка: `glab auth status 2>&1 | grep mindbox.gitlab.yandexcloud.net`. Должно быть `Logged in` и `Token found`. Если нет — остановиться, попросить пользователя залогиниться.
3. **Yandex Tracker MCP подключён.** Проверка: есть ли `mcp__yandex-tracker__*` / tracker-подобный MCP в deferred tools. Если нет — предупредить пользователя, что сценарий 3 не сможет автоматически создать карточку (скилл продолжит работу, но создание карточки попросит сделать вручную).
4. **URL передан.** Если аргумент пустой — спросить у пользователя `Вставь ссылку на автобаг в Sentry` и дождаться ответа.

После успешного preflight кратко отрапортуй пользователю: `Preflight OK: Sentry MCP ✓, glab ✓, Tracker MCP ✓/✗`.

## Шаг 1 — Парсинг URL

Из URL вида `https://autobugs.mindbox.ru/organizations/<org>/issues/<issue_id>/?environment=<env>&project=<project_id>` извлечь:
- `org_slug` (обычно `mindbox`)
- `issue_id` (номер из path)
- `project_id` (query param)
- `environment` (query param, может отсутствовать)

Если URL не распарсился — спросить пользователя.

## Шаг 2 — Сбор контекста из Sentry

Через Sentry MCP, без WebFetch:
1. `mcp__sentry__get_sentry_resource` с полным URL → issue (title, culprit, project slug, first/last seen, count, userCount, tags).
2. `mcp__sentry__list_issue_events` с `organizationSlug` и issue short-id (`<PROJECT-SLUG>-<NUM>`) → последние события со stack trace.
3. При необходимости `mcp__sentry__get_issue_tag_values` по `environment`, `release`, `server_name` — чтобы понять распределение.

Зафиксируй для отчёта: тип ошибки (класс exception + сообщение), repo/service (= Sentry project slug), culprit (файл+строка), stack trace, частота (events + users), окружения, release, первый/последний раз.

## Шаг 3 — Локализация репозитория (строгий порядок)

1. **Имя репо = Sentry project slug.** Пример: project `analytics-settings` → репо `analytics-settings`.
2. **Сначала проверить локально:** `~/Projects/<slug>`.
   - Есть → `git fetch origin`. Далее:
     - Рабочая копия **чистая** на `main` → `git pull --ff-only origin main`.
     - Рабочая копия **грязная** или на чужой ветке → **НЕ трогать**. Работать через `git show origin/main:<path>` для чтения актуального кода. Предупредить пользователя в отчёте.
3. **Если локально нет** — клонировать через glab: `glab repo clone <org-path>/<slug> ~/Projects/<slug>`. Если в GitLab слаг не нашёлся (другой namespace, другой name) — спросить у пользователя путь к репо.

## Шаг 4 — Анализ кода

1. Из stack trace выдели frames из собственного кода (`in_app: true`, не vendor/framework).
2. Прочитай culprit-файл и соседние по цепочке (через `origin/main` если dirty).
3. Определи класс проблемы:
   - **Транзиентный I/O** (5xx, timeout, reset, throttling) → кандидат на сценарий 1.
   - **Мелкий локальный баг** (null, off-by-one, missing case, неправильный формат) → кандидат на сценарий 2.
   - **Продуктовая / архитектурная проблема**, требующая решения команды-владельца → сценарий 3.

## Шаг 5 — Рекомендация

Выбери ровно один сценарий. Обоснуй данными (частота, окружения, класс ошибки, фрагмент кода).

### Сценарий 1 — Ослабить логгирование

**Когда:** ошибка транзиентная, уже ретраится успешно, не требует действий — шумит в Sentry.
**Что предложить:** снизить уровень лог-сообщения (Error → Warn/Info), оставить метрику, сохранить контекст. Предложить diff (без применения, если пользователь не попросил).

### Сценарий 2 — Пофиксить по месту

**Когда:** локальный очевидный баг — фикс в 1-2 файлах, риск низкий.
**Что предложить:** показать проблему + предложенный патч. Создание MR — только после подтверждения пользователем.

### Сценарий 3 — Карточка в трекер + ignore в Sentry

**Когда:** нужна работа команды-владельца, продуктовое решение, рефакторинг.

Действия:
1. **Спросить у пользователя две вещи одним сообщением:**
   - Доску/очередь в Yandex Tracker для карточки (например `HOMEPAGE`, `ANALYTICS`).
   - N дней для ignore в Sentry (если не ответит — дефолт 7).
2. Создать карточку в трекере. Шаблон тела:
   ```
   ## Проблема
   <что происходит, в каком сервисе, в каких окружениях>

   ## Причина
   <root cause из анализа кода, ссылка на файл:строку>

   ## Мотивация
   <частота, влияние на пользователей, почему не silence/not quick-fix>

   ## Автобаг
   <полный URL из Sentry>
   ```
3. `mcp__sentry__update_issue` со `status=ignored` и TTL N дней (поле см. в схеме тулзы).
4. Сообщить пользователю: создана карточка `<key>`, Sentry issue заигнорен на N дней.

Если Tracker MCP недоступен — сформировать тело карточки как текст и попросить пользователя создать вручную, всё остальное (ignore) — выполнить.

## Формат финального отчёта

```
# Триаж <ISSUE-ID>
- Сервис/репо: <slug>
- Ошибка: <тип и сообщение>
- Culprit: <file:line>
- Частота: <events>/<users>, окружения: <list>, first seen: <date>
- Файлы смотрел: <list>
- Рекомендация: Сценарий <1|2|3> — <краткое обоснование>
- Действия: <что сделано / что предложено пользователю>
- Блокеры: <если есть — dirty worktree, нет Tracker MCP, неясен маппинг репо и т.п.>
```

## Common Mistakes

| Ошибка | Что делать |
|---|---|
| Пошёл в Sentry через WebFetch | Всегда Sentry MCP. WebFetch не даёт stack trace и теги. |
| Сделал `git reset`/`git stash` на грязной копии | НЕ трогать чужую работу. Читать через `git show origin/main:<path>`. |
| Применил ignore без карточки | Автобаг вернётся через N дней без прогресса. Ignore — только парой с карточкой в сценарии 3. |
| Придумал N дней сам | Спросить у пользователя. Дефолт 7 только если явно не ответил. |
| Выбрал сценарий 1 для стабильной 4xx | 4xx на конкретных данных — это не транзиент. Иди в сценарий 3. |
| Создал MR в сценарии 2 без подтверждения | Сначала показать patch, получить OK. |
| Забыл поставить ссылку на автобаг в карточке | Тело карточки без ссылки на Sentry — неотслеживаемо. Ссылка обязательна. |

## Red Flags — STOP

- "MCP Sentry не подключён, но я посмотрю через веб" → стоп, preflight fail.
- "Рабочая копия грязная, но я быстро стэшну" → стоп, читай через origin/main.
- "Создам карточку, а доску возьму дефолтную" → стоп, спроси пользователя.
- "Сценарий 2, сразу пушу MR" → стоп, показ patch → подтверждение → MR.
